# Архитектура проекта

## Технологический стек

Проект построен на следующем стеке:

- **React**
- **Vite**
- **TypeScript**
- **Zustand** — глобальное состояние редактора
- **Konva.js** — 2D-рендеринг сцены

---

## Общая идея архитектуры

Agenix сейчас — это **клиентский визуальный редактор сцены и логики**.

На текущем этапе архитектура должна поддерживать:

- работу со сценой и объектами;
- иерархию объектов;
- инспектор свойств;
- текстовую логику как отдельный слой данных;
- сохранение и загрузку сцены;
- безопасный импорт без крашей.

Архитектура **не ориентирована** на полноценный runtime-движок.

---

## Основные архитектурные слои

### 1. UI-слой
Отвечает за:
- layout редактора;
- панели;
- инспектор;
- hierarchy;
- кнопки import/export;
- панель текстовой логики.

### 2. Слой состояния
Построен на Zustand.

Отвечает за:
- объекты сцены;
- выбор объекта;
- выбор правила текстовой логики;
- camera / viewport state;
- actions изменения сцены;
- actions для логики.

### 3. Слой данных сцены
Содержит:
- `GameObject`
- `logicRefs` внутри объекта как старый слой метаданных
- `logicItems` как новый верхнеуровневый слой текстовой логики

### 4. Слой сериализации / валидации
Отвечает за:
- export/import сцены;
- проверку формата данных;
- защиту редактора от битого JSON.

## Главные сущности проекта

## GameObject
Это базовая сущность сцены.

Типичный объект хранит:
- `id`
- `name`
- `type`
- `parentId`
- `childrenIds`
- `tags`
- `transform`
- `properties`
- `logicRefs`
- `description`
- визуальные параметры (`width`, `height`, `color` и т.д.)

### Назначение
`GameObject` описывает сам объект сцены:
- где он находится;
- как выглядит;
- какие у него свойства;
- как он встроен в иерархию.

---

## logicRefs
Это старый слой логических ссылок внутри `GameObject`.

### Важно
`logicRefs` пока сохраняются и **не удаляются**.

### Их роль
- объектные ссылки/метаданные;
- старый уже существующий уровень логики;
- часть обратной совместимости текущей модели.

### Ограничение
Их не нужно путать с новой текстовой логикой.

---

## logicItems
Это новый отдельный слой текстовой логики сцены.

Текущая форма сущности:

```ts
type LogicTextItem = {
  id: string;
  title: string;
  text: string;
  relatedObjectIds: string[];
  enabled: boolean;
  tags: string[];
  notes: string;
};

Назначение

logicItems нужны для:

хранения человекочитаемой логики;
описания сценариев;
описания роли объектов;
связывания одного правила с несколькими объектами.
Важно

logicItems хранятся отдельно от объектов, на уровне сцены.

Сериализация сцены
Историческое состояние

Изначально сцена экспортировалась как:

GameObject[]

То есть экспортировался только массив объектов.

Текущее архитектурное требование

Сцена должна двигаться к формату:

{
  objects: GameObject[];
  logicItems: LogicTextItem[];
}
Почему

Потому что текстовая логика уже стала отдельной частью состояния сцены и не должна жить только в памяти.

Обратная совместимость

Импорт должен поддерживать оба формата:

Старый формат
GameObject[]
Новый формат
{
  objects: GameObject[];
  logicItems: LogicTextItem[];
}
Валидация

Слой валидации нужен для того, чтобы:

не принимать мусор без контроля;
не ронять приложение;
не затирать рабочую сцену невалидным импортом;
частично принимать валидные данные, если это безопасно.
Важное правило

Невалидные objects и logicItems не должны ронять приложение.

Актуальные ограничения архитектуры

На текущем этапе архитектура осознанно не включает:

runtime execution для текстовой логики;
graph editor;
DSL / scripting language;
AI parsing;
cleanup dangling relatedObjectIds;
отдельную сложную event system;
backend persistence;
multi-user / collaboration.
Осознанные упрощения

Проект специально пока остаётся простым:

логика хранится как обычный текст;
связи задаются через ID объектов;
UI не перегружен;
структура данных остаётся плоской и читаемой;
сложные системы не вводятся заранее.

Это не “недостаток”, а осознанный выбор текущего этапа.

Ключевая мысль по архитектуре

Архитектура Agenix сейчас должна обслуживать редактор, а не полноценный игровой движок.

Это значит:

простые данные;
понятные связи;
предсказуемый UI;
безопасная сериализация;
человекочитаемая логика;
минимальная сложность.

Если какое-то решение начинает уводить проект в сторону “сложной движковой архитектуры”, его нужно считать подозрительным, пока это не требуется отдельной фазой.
