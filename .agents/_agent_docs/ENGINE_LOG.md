## [2026-03-30] Phase 9E: Persistence для текстовой логики
**Статус:** Завершено

### Кратко
`logicItems` теперь сохраняются и восстанавливаются вместе со сценой при Export / Import.

### Почему логика терялась до этого
Три причины:
1. `handleExport` в `App.tsx` сериализовал только `JSON.stringify(objects)` — без `logicItems`
2. `validateAndFilterScene` ожидал на вход массив (`Array.isArray(data)`) — не поддерживал объектный формат
3. `setObjects` в store заменял только `objects`, не трогая `logicItems`

### Что было исправлено

#### `src/App.tsx`
- `handleExport` теперь экспортирует `{ objects, logicItems }` вместо голого массива
- `handleImport` извлекает из результата валидации и `objects`, и `logicItems`
- Вместо `setObjects` вызывается новый `setScene`, который атомарно заменяет и objects, и logicItems
- Лог импорта дополнен информацией о logicItems

#### `src/store/useCanvasStore.ts`
- Добавлен action `setScene(objects, logicItems)` — атомарная замена всего состояния сцены с сбросом выделения

#### `src/utils/sceneValidation.ts`
- Добавлена функция `validateLogicItem` — проверяет все обязательные поля LogicTextItem
- Добавлена внутренняя функция `extractSceneData` — определяет формат (старый массив / новый объект) и извлекает objects и logicItems
- `validateAndFilterScene` теперь возвращает `{ objects, logicItems, report }` вместо `{ objects, report }`
- В report добавлены поля `logicTotal`, `logicValid`, `logicDiscarded`
- Поддержка обратной совместимости: старый формат `GameObject[]` импортируется с `logicItems: []`

### Обратная совместимость
- Импорт старого формата (массив объектов) по-прежнему работает, logicItems при этом = []
- Невалидные logicItems отбрасываются поштучно, не ломая весь импорт
- Невалидные objects не затирают текущую сцену

### Ручная верификация
Пройден полный сценарий:
1. Созданы объекты Box и Zone
2. Создано текстовое правило "PersistTest" с текстом и привязкой к объектам
3. Нажат Export — JSON перехвачен программно
4. Подтверждено: JSON содержит ключи `objects` (2 шт) и `logicItems` (1 шт)
5. Программно выполнен Import того же JSON
6. Подтверждено: объекты восстановлены в Hierarchy, правило восстановлено в LogicPanel

## [2026-03-30] Phase 10A: Устранение dangling ссылок логики (Logic Integrity Cleanup)
**Статус:** Завершено

### Кратко
Внедрен механизм автоматической очистки связей текстовой логики с удаленными или отсутствующими объектами, чтобы предотвратить появление "мертвых" ссылок в `relatedObjectIds`.

### Что было исправлено

#### 1. Утечка ID при удалении объектов
Ранее, если объект удалялся из сцены, его ID навсегда оставался в массиве `relatedObjectIds` связанных с ним текстовых правил.
**Решение:** В `src/store/useCanvasStore.ts` действие `removeObject` теперь итерируется по всем `logicItems` и удаляет ID удаляемого объекта из их `relatedObjectIds`.

#### 2. Захват "мертвых" ссылок при импорте
Ранее импорт валидировал сами объекты и сами правила логики, но не гарантировал, что ID объектов внутри `relatedObjectIds` действительно существуют в импортируемой сцене.
**Решение:** В `src/utils/sceneValidation.ts` добавлен шаг `sanitizeLogicItems`. После того как базовые объекты и базовая логика признаны валидными, функция извлекает набор всех импортируемых `id` объектов. Затем она "просеивает" `relatedObjectIds` каждого правила, удаляя те ID, которые не вошли в этот набор. 

#### 3. Визуализация в UI (вспомогательно)
Для того чтобы пользователь мог инициировать удаление объекта (и тем самым проверить cleanup), в `src/components/Inspector.tsx` в правый верхний угол добавлена кнопка **Delete**.

### Как теперь работает cleanup
- **При удалении:** Удаляется сам объект. Все правила логики сохраняются (даже если у них больше нет привязанных объектов), но ID удаленного объекта из них убирается.
- **При импорте:** Если сторонний `scene.json` содержал правило со ссылкой на несуществующий объект, правило загрузится, но битая ссылка будет вычищена до попадания в store. В лог импорта будет выведено предупреждение с количеством вычищенных `relatedObjectIds` (`report.danglingRefsRemoved`).

### Что сознательно НЕ делалось
- Не создавалась сложная система диагностики или "Repair tool" (ограничились санитизацией при импорте и автоматической очисткой при удалении).
- Не писался графический редактор графов для визуализации битых связей.
- Правила без объектов сознательно **не** удаляются, так как могут содержать важные заметки или глобальный сценарий.

### Что именно проверено вручную
- Проверен JS-скриптом сценарий создания Box и правила, связывания их, а затем вызова `store.removeObject(boxId)`. Зафиксировано, что `boxId` исчезает из `logicItems[0].relatedObjectIds`.
- Проверен UI на наличие кнопки "Delete" в Inspector.
- Убедились, что `App.tsx` корректно обрабатывает новый параметр `danglingRefsRemoved` из результата валидации и отображает его в отчете импорта.

Разработан минимальный UI для редактирования классов/шаблонов объектов без перегруженной prefab-системы. Пользователь теперь может физически управлять коллекцией классов прямо в редакторе.

### Что было сделано
- Создан новый компонент `ClassPanel.tsx`. Он похож по стилистике на `LogicPanel.tsx` и работает как соседняя вкладка.
- В нижней панели (`App.tsx`) добавлены кнопки для переключения между "Logic Rules" и "Object Classes".
- `ClassPanel` позволяет:
  - Видеть отфильтрованный список всех классов (с поиском).
  - Создавать новые классы (кнопка "+").
  - Удалять выбранный класс (кнопка с корзиной + подтверждение).
  - Редактировать для выбранного класса: `name`, `baseType`, `description`, `defaultDescription`, `defaultTags` и `defaultProperties`.
- В `Inspector` (добавленный на этапе 11A) выпадающий список существующих `objectClasses` теперь реально работает с созданными из UI классами: класс можно создать в панели снизу и назначить объекту сверху.

### Что сознательно НЕ делалось
- Не создавалась сложная inheritance tree / nodes system для классов (только плоский список).
- Не реализовывались overrides для инспектора, class default properties не применяются автоматически на лету (только заложен фундамент их описания).
- Графическое представление графа или AI-парсер не добавлялись.
- Редизайн приложения или другие крупные изменения не производились.

### Как теперь работает UI классов и назначение:
В нижней панели теперь 2 вкладки. При переключении на "Object Classes" отображается панель редактирования классов (со списком слева и свойствами справа). В инспекторе (Inspector) есть поле выбора "CLASS ID", которое берёт данные о реально существующих классах из global store. При удалении класса из списка (после подтверждения), если он был назначен какому-либо объекту, это назначение обнуляется (`classId` становится `null`, объекты не удаляются и приложение не падает).

- Store `deleteObjectClass` механизм "мягкого" удаления классов с очисткой ссылок в связанном объекте работает корректно (так как был покрыт на этапе 11A и протестирован логически).

### Fix / Hotfix для удалений (Automated testing compatibility)
Браузерная команда `window.confirm` была полностью заменена на встроенный `inline UI state` в самом списке классов. Теперь при нажатии на 🗑️ (корзину) элемент списка визуально меняется на диалог подтверждения с кнопками «Yes» и «No».
Это решило проблему с автоматизированным тестированием (где нативные алерты молча отклонялись) и значительно улучшило UX, не требуя построения тяжеловесной глобальной системы модальных окон.
