## OQ-001 — Что делать с висячими relatedObjectIds после удаления объекта

Сейчас текстовая логика хранит связи с объектами через `relatedObjectIds`.

Если объект удаляется, его ID может остаться внутри `logicItems`.

### Текущее состояние
- это считается допустимым временным ограничением;
- редактор не должен падать из-за таких ссылок;
- автоматическая очистка пока не является обязательной частью текущего этапа.

### Открытый вопрос
Какой подход выбрать позже:

1. **Каскадная очистка при удалении объекта**
   - при удалении объекта автоматически удалять его ID из всех `logicItems`

2. **Мягкая очистка на уровне UI**
   - не хранение чинить, а просто не показывать несуществующие объекты в панели логики

3. **Отдельная команда cleanup**
   - не трогать данные автоматически, но дать кнопку или шаг "очистить битые связи"

### Почему вопрос ещё открыт
На текущем этапе это не блокирует проект, но позже может ухудшить чистоту данных.

## OQ-002 — Нужен ли следующий уровень поверх logicItems: сценарии / группы правил

Сейчас `logicItems` — это отдельные независимые текстовые правила.

### Открытый вопрос
Нужен ли позже ещё один слой группировки, например:

- сценарии;
- flow-группы;
- тематические наборы логики;
- логические блоки по зонам / системам.

### Возможные направления
1. Оставить всё плоским списком `logicItems`
2. Добавить простую группировку по тегам
3. Добавить отдельную сущность уровня `Scenario`
4. Добавить папки / категории без новой модели исполнения

### Почему вопрос открыт
Пока проект маленький, плоский список подходит.  
Когда логики станет много, может понадобиться дополнительная организация.

## OQ-003 — Останется ли логика полностью свободным текстом или станет полуструктурированной

Сейчас основной подход — свободный человекочитаемый текст.

Это правильно для текущего этапа.

### Открытый вопрос
Нужно ли позже постепенно вводить лёгкую структуру поверх текста, например:

- trigger
- action
- target
- condition
- comment

### Возможные варианты
1. Оставить только свободный текст
2. Добавить опциональные поля рядом с текстом
3. Сделать гибрид:
   - основной текст остаётся главным
   - дополнительные поля появляются только как помощь
4. Перейти к жёсткой схеме

### Предпочтительное направление на будущее
Если и добавлять структуру, то только мягко и поверх текста, а не вместо текста.

### Почему вопрос открыт
Сейчас рано превращать систему в мини-язык программирования, но позже может понадобиться лучшая фильтрация и навигация.

Сейчас, если связь битая, это максимум должно безопасно переживаться данными и UI.

### Открытый вопрос
Нужно ли позже визуально показывать проблемные места, например:
- badge "broken link";
- warning icon в LogicPanel;
- warning в Inspector;
- отдельный список проблем при импорте;
- подсветка rule с несуществующими object IDs.

### Почему вопрос открыт
Это может сильно улучшить прозрачность данных, но пока не является обязательным для базового этапа.

## OQ-005 — Нужна ли отдельная фильтрация / поиск по text logic

Сейчас текстовая логика уже имеет:
- title
- text
- tags
- relatedObjectIds
- enabled
- notes

### Открытый вопрос
Когда логики станет больше, что должно появиться раньше:

1. поиск по title / text
2. фильтр по тегам
3. фильтр по объекту
4. фильтр по enabled
5. комбинация всего вышеперечисленного

### Почему вопрос открыт
Для маленького количества правил это не критично. Для большой сцены станет важным.

## OQ-006 — Нужно ли визуально различать старые logicRefs и новую text logic

Сейчас в проекте есть два разных слоя:

### Старый слой
`logicRefs` внутри `GameObject`

### Новый слой
`logicItems` как отдельная текстовая логика сцены

### Открытый вопрос
Нужно ли позже явнее разделить эти две сущности в UI и документации, например:
- более разными названиями;
- разными визуальными блоками;
- подсказками в Inspector;
- отдельными описаниями в агентских документах.

### Почему вопрос открыт
Пока это ещё не ломает использование, но по мере роста проекта смешение терминов может начать путать и пользователя, и агентов.

## OQ-007 — Нужен ли отдельный отчёт о целостности сцены при импорте text logic

Для объектов уже существует валидация и отчётность по проблемам импорта.

### Открытый вопрос
Нужно ли позже расширить отчёт импорта специально для text logic, например:
- сколько `logicItems` импортировано;
- сколько отброшено;
- какие были невалидны;
- какие содержат битые `relatedObjectIds`.

### Почему вопрос открыт
Когда persistence для text logic станет основным рабочим сценарием, прозрачность импорта станет гораздо важнее.

Следующие темы больше не считаются открытыми и не должны снова подниматься агентами как будто они не решены:

- нужен ли сейчас runtime execution

- нужно ли сейчас делать graph view  
  **Уже решено:** нет, не нужен
