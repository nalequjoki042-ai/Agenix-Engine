# Решения по проекту

Этот файл фиксирует решения, которые уже считаются принятыми и не должны заново обсуждаться агентами без явной причины.

---

## D-001 — Текущее направление продукта

**Решение:**  
Agenix сейчас развивается как **визуальный редактор сцены и логики**, а не как runtime-heavy game engine.

**Что это значит:**
Проект должен помогать пользователю:
- размещать и организовывать объекты сцены;
- видеть структуру сцены;
- редактировать свойства объектов;
- понимать, какой объект за что отвечает;
- хранить и редактировать логику в человекочитаемом виде.

**Что сейчас НЕ является приоритетом:**
- runtime execution;
- физика;
- ECS;
- узловое исполнение логики;
- язык скриптов;
- AI-генерация;
- backend / cloud sync.

**Почему:**  
Главная ценность текущего этапа — наглядность, понятность и удобное редактирование смысла сцены, а не симуляция.

## D-002 — Текстовая логика сейчас является данными, а не исполняемым поведением

**Решение:**  
Текстовая логика на текущем этапе — это **описательный слой данных**.

**Что это значит:**
Она нужна для того, чтобы пользователь мог:
- описывать, что должно происходить;
- фиксировать смысловые связи между объектами;
- указывать роль объекта;
- хранить поведение в понятном текстовом виде.

**Следствие:**
- нет исполнения логики в runtime;
- нет update loop для текстовых правил;
- нет парсинга текста в действия;
- нет state machine / trigger system;
- нет автоматического превращения текста в игровой код.

**Почему:**  
Сейчас цель — понятное описание и визуализация логики, а не исполнение.

---

## D-003 — Текстовая логика хранится отдельно от GameObject

**Решение:**  
Текстовая логика хранится **не внутри `GameObject`**, а отдельным верхнеуровневым слоем сцены.

**Принятая модель:**  
Отдельная коллекция `logicItems`, связанная с объектами через ID.

**Почему это решение принято:**
Одно логическое правило может относиться сразу к нескольким объектам.

Примеры:
- кнопка + дверь;
- зона + награда;
- триггер + свет;
- NPC + маршрутная точка.

Если хранить такие правила внутри одного объекта, структура быстро станет запутанной.

---

## D-004 — Старые logicRefs сохраняются и пока не удаляются

**Решение:**  
Существующие `logicRefs` внутри `GameObject` остаются в проекте и пока **не удаляются, не мигрируются и не переосмысляются**.

**Что это значит:**
Сейчас в проекте есть два разных слоя:

### 1. `logicRefs`
Старый объектный слой логических ссылок / метаданных.

### 2. `logicItems`
Новый человекочитаемый слой текстовой логики.

**Почему:**  
Удаление или миграция `logicRefs` прямо сейчас только раздует scope и повысит риск регрессий.

**Правило для агентов:**  
Не удалять `logicRefs`, не переписывать их и не пытаться автоматически объединить их с `logicItems`, если задача этого явно не требует.

## D-005 — Принятая форма сущности текстовой логики

**Решение:**  
Текущая текстовая логика строится вокруг следующей структуры:

```ts
type LogicTextItem = {
  id: string;
  title: string;
  text: string;
  relatedObjectIds: string[];
  enabled: boolean;
  tags: string[];
  notes: string;
};

Назначение полей:

id — стабильный идентификатор;
title — короткое название правила;
text — основное человекочитаемое описание;
relatedObjectIds — список объектов, к которым относится правило;
enabled — активно правило или нет;
tags — дополнительные метки;
notes — дополнительные заметки.

Почему именно так:
Это минимальная, но уже полезная структура:

она поддерживает обычный текст;
позволяет связывать логику с объектами;
не перегружает модель;
даёт место для будущего роста без переусложнения.
D-006 — UI для логики строится по гибридной схеме

Решение:
Текстовая логика в интерфейсе показывается по гибридной схеме.

Глобальное редактирование

Для этого используется отдельная LogicPanel, где можно:

видеть все правила;
выбирать правило;
редактировать текст и поля;
управлять связями с объектами.
Локальная видимость на объекте

В Inspector показываются правила, связанные с текущим выбранным объектом.

Почему выбран именно такой подход:

если логика только в Inspector, она становится слишком фрагментированной;
если логика только в отдельной панели, теряется контекст объекта.

Гибрид даёт и общий обзор, и объектный контекст.

D-007 — Русский свободный текст является нормальным способом ввода

Решение:
Пользователь должен иметь возможность писать текстовую логику обычным человеческим языком, в том числе по-русски.

Примеры допустимого формата:

Если игрок нажимает кнопку, дверь открывается
Зона выдаёт деньги, если условие выполнено
Этот объект отвечает за спавн врагов

Следствие:

обычные input и textarea сейчас достаточны;
не нужен DSL;
не нужен язык скриптов;
не нужен обязательный AI-парсинг;
не нужно заставлять пользователя писать “по-программистски”.

Почему:
Редактор должен помогать формулировать и хранить смысл, а не загонять пользователя в технический синтаксис.

D-008 — Работа по фазам должна идти узкими подэтапами

Решение:
Каждый активный этап делается небольшими изолированными подэтапами, а не одним огромным куском.

Что это значит:
Агент не должен в одной задаче самовольно объединять:

store;
UI;
импорт/экспорт;
валидацию;
cleanup;
новые архитектурные идеи;
будущие фичи.

Почему:
Широкие пачки изменений часто создают ложное ощущение “всё готово”, а потом оказывается, что критичный сценарий руками не работает.

Операционное правило:
CURRENT_TASK.md должен описывать только один активный подэтап.

D-009 — Сцена должна сохранять и восстанавливать text logic

Решение:
Новая текстовая логика должна сохраняться вместе со сценой.

Целевое направление формата сцены:

Старый формат:

GameObject[]

Новый формат:

{
  objects: GameObject[];
  logicItems: LogicTextItem[];
}

Важно:
Импорт должен поддерживать оба формата:

старый массив объектов;
новый объект сцены с objects и logicItems.

Почему:
Иначе текстовая логика остаётся только в памяти и теряется после Export / Import или перезагрузки.

D-010 — Валидация должна быть безопасной, а не агрессивной

Решение:
Импорт и валидация должны вести себя безопасно для пользователя.

Что это значит:

кривые данные не должны ронять приложение;
невалидные logicItems не должны ломать сцену;
частично валидные данные можно принимать частично;
текущая рабочая сцена не должна затираться мусором.

Почему:
Стабильность редактора сейчас важнее, чем “идеальная строгость” схемы.

D-011 — Висячие ссылки допустимы как временное ограничение

Решение:
Если после удаления объекта в relatedObjectIds остаются старые ID, это сейчас считается известным ограничением, а не критической ошибкой.

Что это значит:

каскадная очистка ссылок пока не обязательна;
cleanup не должен раздувать текущий scope;
приоритет важнее у основной функциональности.

Статус:
Известное ограничение, но не блокер текущего этапа.

D-012 — Документация должна отражать реальное состояние кода, а не оптимистичный отчёт

Решение:
Агентские документы должны описывать то, что реально работает, а не то, что “предполагалось завершить”.

Что это значит:
Нельзя считать фазу завершённой, если:

ручной сценарий не пройден;
JSON не содержит нужные данные;
данные живут только в памяти;
UI есть, но ключевой workflow руками не работает.

Почему:
Ложный статус “готово” ломает контекст следующим агентам и приводит к повторным ошибкам.

D-013 — Inspector и LogicPanel должны быть связаны через общий store state

Решение:
Если нужно синхронизировать выбор правила между Inspector и LogicPanel, это делается через общий state в store, а не через локальные несвязанные состояния.

Текущее направление:
Использовать что-то вроде selectedLogicItemId на уровне глобального стора.

Почему:
Иначе пользователь выбирает правило в одном месте, а второе место об этом не знает.

D-014 — Не надо преждевременно превращать текстовую логику в полу-язык программирования

Решение:
На текущем этапе нельзя преждевременно раздувать текстовую логику до сложной схемы вида:

trigger;
condition;
action;
target;
delay;
state;
branches;
nesting.

Почему:
Это быстро превращает редактор логики в мини-конструктор скриптов, а проект пока не к этому идёт.

Разрешённый минимум сейчас:

название;
основной текст;
связанные объекты;
теги;
заметки;
enabled.
Краткая сводка уже принятых решений

Сейчас в проекте зафиксировано следующее:

Agenix — это визуальный редактор сцены и логики.
Текстовая логика — описательные данные, а не исполняемый runtime.
Текстовая логика хранится отдельно от GameObject в logicItems.
Старые logicRefs остаются и не удаляются.
UI логики строится по гибридной модели: LogicPanel + Inspector.
Русский свободный текст — нормальный и основной способ ввода.
Этапы нужно делать узкими подэтапами.
logicItems должны сохраняться вместе со сценой.
Валидация должна быть безопасной.
Висячие ссылки пока допускаются как известное ограничение.
Документация должна отражать фактическое состояние проекта.
Выбор правила должен синхронизироваться через store.
Не нужно превращать текстовую логику в язык программирования раньше времени.

Если какое-то из этих решений изменится в будущем, этот файл нужно явно обновить.
