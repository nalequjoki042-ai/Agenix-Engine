# Skill: Phase Implementer

## Назначение
Этот skill используется, когда нужно реализовать **ровно одну текущую фазу** проекта Agenix по `CURRENT_TASK.md` без расползания scope.

Он нужен для ситуаций, когда агент должен:
- прочитать текущую фазу;
- найти нужные файлы;
- внести минимально достаточные изменения;
- не делать соседние этапы заранее;
- честно зафиксировать, что реально работает.

---

## Когда использовать
Используй этот skill, если задача:
- является активной фазой из `CURRENT_TASK.md`;
- требует правки кода;
- не требует полного переписывания агентских документов;
- должна быть реализована по шагам и в рамках текущего scope.

---

## Обязательный порядок работы

### 1. Сначала прочитать контекст
Перед любыми изменениями обязательно прочитай:

1. `.agents/workflows/CURRENT_TASK.md`
2. `.agents/_agent_docs/DECISIONS.md`
3. `.agents/_agent_docs/PROJECT_DIRECTION.md`
4. `.agents/_agent_docs/ARCHITECTURE.md`
5. `.agents/_agent_docs/ENGINE_LOG.md`

Если задача спорная или затрагивает неоднозначные места:
- `.agents/_agent_docs/OPEN_QUESTIONS.md`

Если задача связана с import/export, validation, broken JSON, cleanup, class links, parentClassId, relatedObjectIds:
- `.agents/_agent_docs/DEV_HARNESS.md`

---

## Главные правила

### Делать только текущую фазу
Если в `CURRENT_TASK.md` указана конкретная фаза/подэтап:
- не делай следующий этап заранее;
- не добавляй будущие фичи;
- не делай “заодно” архитектурные улучшения.

### Не спорить с уже принятыми решениями
Если что-то зафиксировано в `DECISIONS.md`, не переоткрывай это как “open question”.

### Не врать про готовность
Нельзя писать, что этап завершён, если:
- критичный сценарий не проверен;
- UI не прокликан;
- JSON/импорт/экспорт не подтверждены;
- есть только логическая уверенность, но нет фактической проверки.

### Если уверенность неполная
Если что-то не подтверждено:
- явно пометь это как `POSSIBLE_BUG`, `NOT_FULLY_VERIFIED` или `NEEDS_MANUAL_CHECK`;
- кратко напиши, что именно осталось сомнительным;
- не блокируй выполнение безопасной части задачи.

---

## Как реализовывать фазу

### Шаг 1
Понять, какой именно результат требуется по `CURRENT_TASK.md`.

### Шаг 2
Найти минимальный набор файлов, которые реально нужно изменить.

### Шаг 3
Сделать минимально достаточную реализацию без соседних “улучшений”.

### Шаг 4
Проверить:
- не сломался ли существующий flow;
- не произошёл ли unwanted overwrite данных;
- не расползся ли scope.

### Шаг 5
Если возможно, прогнать живую проверку:
- через UI
- через Dev JSON harness
- через консоль
- через клики и проверку реального поведения

### Шаг 6
Только после этого обновлять:
- `.agents/_agent_docs/ENGINE_LOG.md`
- `.agents/workflows/CURRENT_TASK.md`  
и только если фаза реально подтверждена.

---

## Как выбирать способ проверки

### Если задача про UI
Предпочитать:
- реальные клики;
- реальные переключения;
- реальные сценарии пользователя.

### Если задача про import/export или broken data
Предпочитать:
- `Dev JSON`
- `Load Current Scene`
- правку JSON в textarea
- `Apply JSON`

### Если задача касается store/data model
Проверять:
- не ломается ли UI;
- корректно ли данные восстанавливаются;
- не теряются ли значения после переключений / импорта / экспорта.

---

## Что не делать

Не делать без прямой задачи:
- runtime execution;
- graph view;
- ECS;
- physics;
- AI parsing;
- scripting language;
- prefab system;
- override UI;
- inheritance tree, если задача не про inheritance;
- большой debug framework;
- редизайн редактора.

Не удалять:
- старые `logicRefs`
если это не требует активная фаза.

---

## Что считать хорошей реализацией
Хорошая реализация:
- простая;
- безопасная;
- не затирает данные неожиданно;
- понятна непрограммисту;
- реально работает;
- подтверждена проверкой.

Плохая реализация:
- “умная”, но не проверенная;
- ломает старые сценарии;
- раздувает scope;
- тянет будущие системы раньше времени;
- завершает фазу только “на словах”.

---

## Как оформлять результат

После выполнения отвечай строго в таком формате:

1. Что было сделано
2. Какие файлы изменены
3. Что теперь работает
4. Что сознательно НЕ делалось
5. Риски / открытые моменты
6. Что именно проверено вручную
7. Что НЕ подтверждено, если такое осталось

---

## Как работать с `.agents`
Не переписывай `.agents` как отдельную задачу.

Разрешены только минимальные обязательные изменения:
- `.agents/_agent_docs/ENGINE_LOG.md`
- `.agents/workflows/CURRENT_TASK.md`

Все остальные агентские документы трогать только если это отдельная явная задача.

---

## Process hygiene

If implementation requires `npm run dev`, test runners, or extra terminal processes:
- reuse existing running processes where possible
- avoid starting duplicate dev servers
- stop temporary processes after verification
- if a process remains running intentionally, explicitly say so in the report

Always include in the final report:
- started process or not
- still running or stopped
- relevant port if applicable

---

## Специальное правило для Agenix
Если задача связана с классами, text logic, import/export, validation или inheritance:
- предпочтение всегда отдаётся **понятному и безопасному поведению**,
- а не агрессивной “магии”.

Если есть выбор между:
- сложной, но эффектной системой
- простой, понятной и безопасной

выбирай второе.

---

## Ключевая мысль
Твоя задача — не просто “написать код”, а **дотянуть текущую фазу до реально рабочего состояния и честно это подтвердить**.
