# Правила для агентов

Этот файл задаёт обязательные рабочие правила для любого агента, который выполняет задачи внутри проекта Agenix.

Цель правил:
- не терять контекст проекта;
- не уходить в лишний scope;
- не ломать уже принятые решения;
- не тратить токены и время на повторное переизобретение уже решённых вещей.

---

## 1. Главный принцип работы

Агент должен работать **от текущего состояния проекта**, а не от своих общих предположений о том, как "обычно" строят движки, редакторы или игровые системы.

Перед началом любой задачи агент обязан:
1. прочитать актуальные агентские документы;
2. понять активный подэтап;
3. не выходить за его рамки;
4. не подменять продуктовый вектор своими идеями.

---

## 2. Обязательный порядок чтения перед работой

Перед выполнением любой задачи агент обязан прочитать документы **в таком порядке**:

### Шаг 1 — текущая задача
- `.agents/workflows/CURRENT_TASK.md`

Это главный документ для текущего подэтапа.  
Если агент игнорирует его, он почти наверняка начнёт делать лишнее.

### Шаг 2 — направление проекта
- `.agents/_agent_docs/PROJECT_DIRECTION.md`

Этот файл объясняет, каким проект должен быть в продуктном смысле.

### Шаг 3 — уже принятые решения
- `.agents/_agent_docs/DECISIONS.md`

Этот файл запрещает агенту заново "решать" уже закрытые вопросы.

### Шаг 4 — история этапов
- `.agents/_agent_docs/ENGINE_LOG.md`

Этот файл нужен, чтобы понять:
- что уже было сделано;
- какие фазы уже завершены;
- на каком этапе проект реально находится.

### Шаг 5 — открытые вопросы
- `.agents/_agent_docs/OPEN_QUESTIONS.md`

Этот файл нужен только для реально открытых вопросов, а не для пересмотра зафиксированных решений.

### Шаг 6 — архитектура
- `.agents/_agent_docs/ARCHITECTURE.md`

Этот файл нужен, чтобы понимать:
- какие основные слои уже есть;
- какие файлы важны;
- где что хранится.

---

## 3. Что агенту запрещено делать по умолчанию

Если задача этого явно не требует, агенту запрещено:

- уводить проект в runtime-heavy engine;
- добавлять physics;
- добавлять ECS;
- добавлять graph view / node editor;
- добавлять AI features;
- добавлять scripting / DSL;
- перепридумывать модель логики;
- удалять старые `logicRefs`;
- мигрировать старые данные без явного указания;
- делать крупный рефактор "на будущее";
- расширять scope за пределы активного подэтапа.

---

## 4. Что агент должен считать уже решённым

Следующие вещи не считаются открытыми вопросами и не должны переобсуждаться без явной причины:

1. Проект сейчас — **визуальный редактор сцены и логики**, а не полноценный игровой движок.
2. Текстовая логика сейчас — **данные**, а не runtime execution.
3. Текстовая логика хранится **отдельно от `GameObject`** как `logicItems`.
4. Старые `logicRefs` пока остаются и не удаляются.
5. UI логики строится по гибридной модели:
   - `LogicPanel`
   - `Inspector`
6. Русский свободный текст — допустимый и желательный способ ввода.
7. Работа должна идти узкими подэтапами, а не гигантскими пачками.

Если агент снова начинает спорить с этими решениями, он действует неправильно.

---

## 5. Как агент должен понимать scope

### Активный подэтап — это жёсткая граница
Если в `CURRENT_TASK.md` указан один конкретный подэтап, агент обязан делать **только его**.

### Нельзя делать "заодно"
Запрещено:
- делать соседние этапы заранее;
- добавлять будущие улучшения;
- пытаться "сразу всё привести в порядок";
- менять unrelated части кода.

### Разрешено только:
- необходимое изменение внутри текущего подэтапа;
- минимальные вынужденные адаптации рядом, если без них задача физически не работает.

---

## 6. Как агент должен принимать решения

Перед любым изменением агент должен проверить:

1. Это действительно входит в текущий подэтап?
2. Это уже не решено в `DECISIONS.md`?
3. Это не уводит проект в сторону движка?
4. Это не раздувает архитектуру без необходимости?
5. Это помогает текущей цели проекта, а не какой-то будущей гипотетической системе?

Если на любой из этих вопросов нет уверенного "да", изменение подозрительное и скорее всего лишнее.

---

## 7. Правила работы с кодом

Агент должен:
- уважать существующую структуру проекта;
- делать минимально достаточные изменения;
- не переписывать большие куски без необходимости;
- сохранять текущий стиль проекта;
- не ломать уже существующее поведение без явной причины;
- не делать "умные" архитектурные перестройки ради красоты.

### Особенно важно
Если задача касается:
- store,
- import/export,
- validation,
- logicItems,
- Inspector,
- LogicPanel,

то агент обязан сначала проверить, как эти вещи уже устроены в коде, а не строить новые параллельные механизмы.

---

## 8. Правила работы с документами `.agents`

Если задача касается агентских документов, агент должен:

- синхронизировать их с реальным кодом;
- не писать оптимистичную ложь;
- не помечать фазу завершённой, если ручной сценарий реально не пройден;
- не оставлять в `OPEN_QUESTIONS.md` уже решённые вопросы;
- не держать `CURRENT_TASK.md` на старом подэтапе, если код уже ушёл дальше;
- не делать `DECISIONS.md` пустым, если решения уже приняты.

---

## 9. Правила для import/export и валидации

Если задача касается импорта, экспорта или валидации, агент должен помнить:

- редактор не должен падать на мусорных данных;
- валидная текущая сцена не должна затираться невалидным импортом;
- частично валидные данные можно принимать частично, если это безопасно;
- обратная совместимость старых форматов важна;
- нельзя считать persistence завершённой, если ручной сценарий не подтверждён.

### Критичное правило
Если после Export / Import данные теряются, этап нельзя считать завершённым даже если код "выглядит правильно".

---

## 10. Правила для text logic

Если задача касается текстовой логики, агент должен помнить:

- `logicItems` — это отдельный слой сцены;
- это не runtime execution;
- это не scripting system;
- это не graph-based logic;
- это не AI feature;
- это человекочитаемая логика и смысловые связи.

### Следствие
Нельзя преждевременно раздувать модель до:
- trigger engine;
- action pipeline;
- state machine;
- scripting language;
- visual behavior graph.

---

## 11. Формат ответа агента после работы

После выполнения задачи агент обязан отвечать в строгом формате:

### 1. Что было сделано
Коротко и по сути.

### 2. Какие файлы изменены
Списком, с пояснением роли каждого файла.

### 3. Что теперь работает
Только фактически работающий результат.

### 4. Что сознательно НЕ делалось
Чтобы было видно границы выполненного scope.

### 5. Риски / открытые моменты
Только реальные, а не надуманные.

---

## 12. Что считается плохим ответом агента

Плохой ответ — это когда агент:
- пишет расплывчато;
- не говорит, какие файлы реально изменил;
- объявляет задачу завершённой без проверки руками;
- уходит в будущее вместо текущего этапа;
- выдумывает несуществующие ограничения;
- смешивает уже сделанное и то, что только предполагается;
- пишет, что "всё готово", когда критичный сценарий падает.

---

## 13. Что считается хорошим поведением агента

Хорошее поведение агента выглядит так:

- сначала читает документы;
- понимает активную задачу;
- не спорит с уже принятыми решениями;
- не расширяет scope;
- меняет только нужные файлы;
- честно фиксирует, что реально работает;
- отдельно указывает, что ещё не сделано;
- не тратит токены на лишние направления.

---

## 14. При конфликте источников

Если агент видит конфликт между:
- кодом,
- `CURRENT_TASK.md`,
- `ENGINE_LOG.md`,
- `OPEN_QUESTIONS.md`,
- `DECISIONS.md`,

он должен действовать так:

### Приоритет источников
1. **Текущий код**
2. **CURRENT_TASK.md**
3. **DECISIONS.md**
4. **PROJECT_DIRECTION.md**
5. **ENGINE_LOG.md**
6. **OPEN_QUESTIONS.md**

### Что это значит
- если код уже ушёл дальше, чем `ENGINE_LOG.md`, нельзя слепо верить старому логу;
- если вопрос уже решён в `DECISIONS.md`, нельзя снова считать его открытым;
- если `CURRENT_TASK.md` противоречит коду, агент должен аккуратно отметить это и не делать ложных выводов.

---

## 15. Главное правило на практике

Агент должен помогать проекту становиться:

- понятнее;
- устойчивее;
- ближе к текущей цели;
- чище по контексту;
- без лишнего “движкового” мусора.

Если изменение этого не делает — скорее всего его не нужно делать сейчас.
