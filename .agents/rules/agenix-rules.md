# Agenix Rules

Этот файл задаёт базовые продуктовые и проектные рамки Agenix Engine.

Он нужен для того, чтобы любой агент сразу понимал:
- что это за проект;
- куда он развивается;
- что уже считается важным;
- чего сейчас делать не нужно.

---

## 1. Что такое Agenix сейчас

Agenix на текущем этапе — это **визуальный редактор сцены и логики**.

Проект предназначен для того, чтобы пользователь мог:
- собирать сцену из объектов;
- видеть структуру сцены;
- редактировать свойства объектов;
- хранить и понимать смысловые связи;
- описывать логику обычным текстом;
- связывать логику с объектами сцены;
- сохранять и загружать результат.

### Ключевая мысль
Agenix сейчас — это не "движок, который всё исполняет", а "редактор, который помогает всё видеть, организовывать и описывать".

---

## 2. Текущее продуктовое направление

На ближайших этапах проект должен усиливаться в следующих направлениях:

### A. Scene Editing
- создание объектов;
- редактирование transform;
- иерархия;
- выбор объектов;
- наглядность сцены.

### B. Object Understanding
- теги;
- description;
- custom properties;
- object-level metadata;
- inspector как центральная точка редактирования объекта.

### C. Human-Readable Logic
- логика в человекочитаемом виде;
- русский свободный текст как нормальный режим ввода;
- хранение логики как данных;
- смысловые связи между логикой и объектами.

### D. Safe Persistence
- export/import сцены;
- безопасная валидация;
- устойчивость к мусорным данным;
- обратная совместимость форматов.

---

## 3. Что сейчас считается приоритетом

Текущие приоритеты проекта:

### Приоритет 1 — Editor Core
- objects
- hierarchy
- inspector
- scene editing
- text logic layer
- export/import

### Приоритет 2 — Понятность
- ясный UI
- прозрачные данные
- логичные связи
- удобное редактирование

### Приоритет 3 — Надёжность
- безопасный импорт
- отсутствие падений
- предсказуемое поведение
- честное состояние документации

---

## 4. Что сейчас НЕ является приоритетом

Следующие направления **не должны** становиться центром проекта на текущем этапе:

- runtime execution engine
- physics
- ECS
- graph editor / node system
- scripting language
- AI generation
- AI parsing of logic text
- backend / cloud sync
- multiplayer / collaboration
- advanced animation systems
- procedural systems
- complex event engine

Если агент без прямой задачи двигает проект в эти стороны, он уходит от текущего продукта.

---

## 5. Как нужно понимать "логику" в Agenix

В Agenix сейчас есть два слоя логики:

### 1. Старый слой: `logicRefs`
Это объектные логические ссылки / метаданные, встроенные в `GameObject`.

### 2. Новый слой: `logicItems`
Это отдельная текстовая логика сцены:
- человекочитаемая;
- связанная с объектами по ID;
- редактируемая через UI;
- не исполняемая в runtime на текущем этапе.

### Ключевое правило
Текстовая логика сейчас — это **описание и структура**, а не движок поведения.

---

## 6. Что сейчас считается правильной моделью логики

Правильная модель текущего этапа:

- логика хранится как данные;
- логика может относиться к нескольким объектам;
- логика редактируется человеком вручную;
- логика остаётся понятной без знания программирования;
- логика показывается и через отдельную панель, и через Inspector;
- логика не требует DSL;
- логика не требует AI;
- логика не требует runtime execution.

---

## 7. Текущая модель пользователя

Проект должен быть полезен человеку, который хочет:

- визуально понимать свою сцену;
- помечать, какой объект за что отвечает;
- писать логику по-русски;
- хранить смысл поведения без необходимости писать код;
- проектировать структуру игры / сцены до стадии runtime.

То есть Agenix сейчас помогает:
- мыслить;
- проектировать;
- описывать;
- организовывать.

---

## 8. Какой UX считается правильным

Хороший UX в Agenix должен:

- быть понятным;
- не перегружать пользователя;
- показывать связи между объектами и логикой;
- помогать быстро редактировать сцену;
- поддерживать обычный текстовый ввод;
- не заставлять пользователя мыслить как движок.

Плохой UX:
- перегружает интерфейс системами "на будущее";
- прячет важные данные;
- превращает логику в псевдокод;
- заставляет пользователя работать как программист, хотя задача редакторская.

---

## 9. Какой подход к архитектуре считается правильным

На этом этапе нужно предпочитать:

- простые модели данных;
- понятные слои состояния;
- минимально достаточный UI;
- безопасную сериализацию;
- узкие подэтапы вместо больших “комбайнов”.

### Неправильный подход
- добавлять сложную движковую архитектуру заранее;
- делать тяжёлые абстракции без пользы;
- строить систему "на потом", когда текущий use case ещё не закрыт.

---

## 10. Что агент должен считать уже установленным курсом

Агент не должен снова обсуждать как открытые темы следующие вещи:

- проект не идёт сейчас в runtime-heavy engine;
- текстовая логика хранится отдельно от `GameObject`;
- старые `logicRefs` пока остаются;
- логика редактируется через `LogicPanel` и видна через `Inspector`;
- обычный русский текст является валидным способом описания логики;
- работа должна идти поэтапно и узко;
- persistence для text logic — обязательный следующий шаг, если он ещё не завершён фактически.

---

## 11. Главное правило про scope

Если задача узкая, агент должен делать **только узкую задачу**.

Нельзя:
- “заодно” делать соседние этапы;
- расширять модель без запроса;
- придумывать новые подсистемы;
- смешивать продуктовую доработку с глубокой архитектурной перестройкой;
- считать, что если есть время, то можно заодно навесить улучшения.

Для Agenix это особенно важно, потому что проект уже проходил через ситуацию, когда “большой полезный пакет” выглядел завершённым, но реальный пользовательский сценарий не работал.

---

## 12. Что считается хорошим развитием проекта

Правильное развитие Agenix выглядит так:

- сцена становится понятнее;
- логика становится видимее и удобнее;
- связи между объектами и логикой читаются лучше;
- данные становятся устойчивее;
- import/export становятся надёжнее;
- UI растёт без превращения в перегруженный комбайн.

---

## 13. Что считается плохим развитием проекта

Неправильное развитие Agenix выглядит так:

- проект уходит в runtime execution слишком рано;
- появляется тяжёлый graph editor без реальной необходимости;
- логика превращается в сложный псевдокод;
- документация перестаёт соответствовать коду;
- агент спорит с уже принятыми решениями;
- текущий подэтап не закрыт, а агент уже строит следующий уровень сложности.

---

## 14. Какой результат должен давать любой следующий этап

Любой хороший следующий этап должен хотя бы в одном из направлений делать проект лучше:

- понятность сцены;
- читаемость логики;
- редактируемость данных;
- связность UI;
- устойчивость сериализации;
- честность и актуальность агентских документов.

Если этап не усиливает ничего из этого, он, скорее всего, не нужен прямо сейчас.

---

## 15. Финальная формула проекта на текущем этапе

Agenix сейчас — это:

**визуальный редактор сцены + удобный редактор человекочитаемой логики + безопасное сохранение структуры проекта**

а не:

**runtime engine + scripting platform + node execution framework**

Любой агент должен отталкиваться именно от этой формулы.
