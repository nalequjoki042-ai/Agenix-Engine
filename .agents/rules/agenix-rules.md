# Agenix Rules / Project Context

## Что это за проект
Agenix Engine — это редактор / платформа для разработки игр с упором на удобную работу с объектами, их свойствами, связями и логикой прямо внутри редактора.

Ключевая идея проекта: не просто вручную программировать поведение объектов, а построить систему, где значительная часть работы с объектами, их структурой и логикой выполняется через удобный редактор.

---

## Текущая стадия проекта
Проект больше не нужно трактовать как путь к runtime-heavy engine.
Текущий вектор развития: визуальный редактор сцены и логики. 
Логика должна быть человекочитаемой и редактироваться простым текстом, в том числе по-русски. 
На текущем этапе логика НЕ исполняется, а хранится исключительно как слой данных.

Сейчас уже есть базовый фундамент:
- React + TypeScript + Vite;
- canvas / viewport;
- базовая работа с объектами;
- заготовки типов объектов;
- общая UI-основа редактора.

Сейчас задача — не расширяться вширь и не уходить в runtime/ECS/physics/graph/AI, а собрать сильный фундамент.

---

## Что уже есть
На текущий момент в проекте уже присутствуют или начаты следующие части:

- центральная рабочая область / canvas;
- базовая сцена;
- создание и отображение объектов;
- стартовые типы объектов (box, zone, unit);
- общее направление интерфейса в сторону editor layout (сцена в центре, hierarchy слева, inspector справа).

---

## Главная идея проекта
Проект должен стать удобной платформой для разработки игр, где:

- в центре находится рабочая сцена / viewport;
- слева располагается hierarchy объектов;
- справа — inspector выбранного объекта;
- объект можно создавать, редактировать, связывать с другими объектами;
- объекту можно задавать тип, класс, теги, родителя, свойства и текстовые сценарии/логику;
- архитектура должна быть понятной, расширяемой и удобной для дальнейшего развития.

---

## Главный фокус текущего этапа
Текущий приоритет проекта:

- текстовая логика / сценарии как слой данных (человекочитаемая логика);
- сохранение логики и привязка к объектам;
- strong editor foundation;
- object model;
- hierarchy;
- inspector;
- parent / child связи;
- tags.

Главная цель сейчас:
**сделать слой текстовой логики как данные, не уводя проект в runtime исполнение.**

---

## Что сейчас НЕ является приоритетом
На текущем этапе намеренно НЕ уходим в следующие направления:

- runtime исполнение (ECS, physics и т.д.);
- AI-слой;
- AI-генерация логики;
- граф зависимостей / мегаграф;
- нодовая визуальная система;
- сложная автоматизация через агентов;
- попытка сразу собрать весь движок целиком.

---

## Почему это важно
Главный риск проекта — слишком рано смешать:

- редактор сцены;
- архитектуру объектов;
- систему логики;
- AI-слой;
- граф и сложные визуальные системы.

Если смешать всё сразу, фундамент станет слабым и проект быстро уйдёт в хаос.

Поэтому текущий курс такой:

1. editor core  
2. object model  
3. текстовая логика как данные
4. только потом будущие расширения (включая runtime/AI)

---

## Ближайшая цель
Ближайшая цель проекта:
сделать слой текстовой логики, чтобы пользователь мог писать логику текстом (в том числе по-русски) для объектов и сохранять её как данные.

# Product Vision / V1 Scope

## Что мы делаем в v1
Первая версия Agenix Engine — это фундамент редактора сцены, а не полноценный игровой движок.

Цель v1:
собрать удобный, понятный и расширяемый editor core, в котором можно создавать объекты, видеть их структуру, редактировать свойства и писать текстовую логику для них как данные.

---

## Что обязательно входит в v1

### 1. Scene / Viewport
- центральная рабочая область;
- отображение сцены;
- базовая работа с объектами на сцене;
- выбор объекта.

### 2. Object Creation
- создание объекта через UI;
- стартовые типы объектов:
  - box
  - zone
  - unit
  - custom (по необходимости позже).

### 3. Hierarchy
- список объектов слева;
- отображение parent / child структуры;
- выбор объекта из hierarchy.

### 4. Inspector
- свойства выбранного объекта справа;
- редактирование текстовой логики / правил.

### 5. Object Model Foundation
- единая структура хранения объектов;
- поддержка parent / child;
- поддержка tags;
- поддержка текстовой логики (хранение как данные).

---

## Что НЕ входит в v1
Следующие вещи не являются частью первой версии и сознательно откладываются:

- runtime-исполнение логики (ECS, physics);
- AI-слой;
- граф зависимостей;
- нодовая логическая система;
- сложное наследование классов;
- автоматическая работа агентов внутри редактора;
- попытка сразу сделать “полноценный next-gen engine”.

---

## Критерий успеха v1
v1 считается успешной, если редактор уже позволяет:

- создать объект;
- увидеть его в hierarchy;
- выбрать его;
- увидеть и изменить его свойства в inspector;
- задать parent / child связи;
- назначить теги;
- написать и сохранить текстовую логику для объекта;
- подготовить объектную модель для будущего расширения.

---

## Главный принцип v1
Сначала:
- editor core;
- object model;
- понятный UI;
- расширяемый фундамент.

Текущий следующий шаг:
- текстовая логика как описательный слой данных (без runtime и AI).

Потом:
- исполнение логики;
- AI;
- graph;
- сложные надстройки.

# Core Entities / Data Model Foundation

## Зачем нужен этот блок
Этот блок фиксирует минимальную модель сущностей проекта.

Цель:
не придумать “идеальную систему на 5 лет вперёд”, а определить такой фундамент, которого уже хватит для:
- scene editor;
- hierarchy;
- inspector;
- parent / child связей;
- tags;
- базовых свойств;
- хранения текстовой логики как данных.

---

## Главные сущности v1

В первой версии достаточно зафиксировать 3 основные сущности:

1. Object Instance  
2. Object Class / Template  
3. Logic Reference / Text Logic

---

## 1. Object Instance
Object Instance — это конкретный объект на сцене.

Примеры:
- box на сцене;
- zone на сцене;
- unit на сцене.

Каждый объект должен хранить минимум:

- `id` — уникальный идентификатор;
- `name` — имя объекта;
- `type` — базовый тип объекта (`box`, `zone`, `unit` и т.д.);
- `className` — класс / шаблон, если используется;
- `parentId` — родительский объект или `null`;
- `childrenIds` — список дочерних объектов;
- `tags` — набор тегов;
- `transform` — базовые координаты и параметры позиции;
- `properties` — расширяемые свойства объекта;
- `logicRefs` — ссылки на связанную логику;
- `description` — описание объекта для редактора.

### Минимальный смысл Object Instance
Object Instance отвечает на вопрос:
**что это за объект, где он находится, кому принадлежит и какие у него свойства.**

---

## 2. Object Class / Template
Object Class / Template — это шаблон, по которому создаются объекты.

Примеры:
- `Zone`
- `Unit`
- `NPC`
- `TriggerZone`
- `Box`

Class / Template нужен для того, чтобы:
- задавать стартовые свойства;
- задавать допустимые теги;
- задавать допустимые типы логики;
- в будущем расширять систему без хаоса.

### Важный принцип
В v1 class/template — это **не сложная наследуемая магия**, а просто удобный шаблон.

---

## 3. Logic Reference / Text Logic
Logic Reference — это ссылка объекта на некоторую логику или само текстовое правило.

Пока в v1 логика не должна быть сложной системой.
Нам нужно только зафиксировать, что объект может быть связан с текстовой логикой (которая просто хранится как текст и не исполняется).

### Что хранит Logic Reference
Минимально:

- `id`
- `name`
- `type` — например:
  - `server`
  - `client`
  - `shared`
- `description`
- `enabled`
- само текстовое содержание логики (если хранится внутри)

### Важный принцип
В v1 логика хранится как **связь / ссылка или текст**, а не как исполняемый код в runtime.

---

## Базовые правила модели данных

### 1. Parent / Child
Связи объектов должны быть явными:
- у объекта есть `parentId`;
- у объекта есть `childrenIds`.

Не хранить родительство “только визуально”. Это часть реальной модели.

### 2. Tags
Tags — это простой набор строковых меток.

В v1 tags нужны:
- для организации;
- для фильтрации;
- для будущего расширения.

Tags пока не должны превращаться в отдельную сложную подсистему.

### 3. Properties
`properties` — это расширяемый словарь свойств объекта.

Он нужен, чтобы:
- не зашивать всё намертво в базовую структуру;
- позволить разным типам объектов хранить свои данные;
- не ломать архитектуру при росте проекта.

### 4. Transform
Минимальный `transform` в v1:
- position x
- position y
- при необходимости z позже
- rotation
- scale

Даже если часть этого пока не используется, лучше сразу предусмотреть нормальный контейнер под transform.
